后端编码规范

声明：

本文档为团队共创，文档中所涉及的技术方案、工具使用方法等信息，均基于当前主流技术环境整理与总结。由于技术更新迭代迅速，以及实际应用环境存在差异，仅供技术学习、参考和内部交流使用，针对最新立项项目，后端开发组对开放、测试、生产环境架构做如下说明。

为保证文档内容的时效性和准确性，若有疑问或建议，可直接联系本人，及时补充更新。

1. 接口设计原则

SOLID 是面向对象设计中非常重要的五个原则，确保系统设计的灵活性和可扩展性：

  • S - 单一职责原则：每个类应该只有一个职责，避免类承担多个不相关的功能。
  • O - 开放封闭原则：系统应该对扩展开放，对修改封闭。也就是说，不应该修改已有的代码，而是通过扩展来实现新的功能。
  • L - 里氏替换原则：子类应该能够替代父类，并且父类的实例应该能够无缝地被子类实例所替代。
  • I - 接口隔离原则：客户端不应该依赖它不需要的接口。接口应该尽量小，功能单一。
  • D - 依赖倒置原则：高层模块不应该依赖低层模块，二者应该依赖于抽象。抽象不应该依赖于细节，细节应该依赖于抽象。

SOLID原则应用示例：

1. 单一职责原则（SRP）：

通俗解释：每个类或模块应该只做一件事，只有一个原因去修改它。

举例：

  • 假设你有一个 UserService 类，它既负责用户注册、登录，又负责发送欢迎邮件和短信通知。这样做违反了单一职责原则，因为它承担了多个不相关的功能。
  • 应该如何做：将发送邮件和短信通知的功能提取到独立的服务中，例如 MailService 和 SmsService，这样 UserService 只负责用户相关的逻辑。

避免做什么：

  • 不要让一个类或模块负责多种功能（如“用户管理 + 邮件发送 + 数据分析”）。
  • 如果一个类在多次修改时有多个原因，那它就违反了 SRP。

2. 开放封闭原则（OCP）：

通俗解释：系统应该对扩展开放，对修改封闭。也就是说，已有的代码应该保持不变，新功能应该通过扩展现有系统来实现。

举例：

  • 假设你有一个 PaymentProcessor 类，它处理不同的支付方式。你现在新增了一个 PayPal 支付方式，如果修改现有的 PaymentProcessor 类来适应新的支付方式，会违反开放封闭原则。
  • 应该如何做：你可以通过继承或接口扩展来实现。例如，定义一个支付接口 PaymentGateway，然后为每种支付方式实现这个接口，PaymentProcessor 类通过调用不同的支付方式来处理支付。

避免做什么：

  • 不要直接修改已有的类（特别是公共类或核心业务代码）。
  • 不要在现有代码中硬编码新的业务逻辑，应该通过扩展来增加新功能。

3. 里氏替换原则（LSP）：

通俗解释：子类应该能够替代父类，并且父类的实例应该能够无缝地被子类实例所替代。

举例：

  • 假设你有一个 Shape 类，子类 Circle 和 Rectangle。如果 Circle 类的 area() 方法返回值和 Shape 类不一致（比如负值），或者 Rectangle 类调用时会抛出异常，那么就违反了里氏替换原则。
  • 应该如何做：确保所有子类都能遵循父类的约定，方法的行为和父类一致。

避免做什么：

  • 不要让子类的行为与父类不兼容，特别是在继承时要确保接口一致。
  • 不要破坏父类的预期行为（如父类返回正数，子类返回负数，或者让父类的逻辑不再有效）。

4. 接口隔离原则（ISP）：

通俗解释：接口应该细粒度和单一，客户端不应该依赖它不需要的接口。

举例：

  • 假设你有一个 PaymentService 类接口，它同时包含了支付、退款、查询交易等多个方法。如果某些类只需要支付功能，但却被迫实现退款和查询交易的功能，就违反了接口隔离原则。
  • 应该如何做：将接口拆分成多个更小的接口，每个接口只包含相关的方法。例如，创建 PaymentInterface，RefundInterface 和 TransactionInterface。

避免做什么：

  • 不要让一个接口包含太多功能，不要强迫用户去实现自己不需要的接口方法。
  • 不要设计一个“一揽子”接口，确保每个接口都是面向特定职责的。

5. 依赖倒置原则（DIP）：

通俗解释：高层模块不应该依赖低层模块，二者应该依赖于抽象（接口）。也就是说，我们不应该让高层的代码直接依赖于低层的实现，而是依赖于抽象的接口或类，这样可以避免紧耦合。

举例：

  • 假设你有一个 OrderService 类，它直接依赖于 MySQLDatabase 类来保存订单数据。如果数据库换成了 MongoDB，那么你就必须修改 OrderService 类，违反了依赖倒置原则。

  • 应该如何做：你可以引入一个数据库接口（如 DatabaseInterface），然后 OrderService 依赖于这个接口，而不直接依赖具体的数据库实现。

3. 设计模式应用

控制反转与依赖注入（IoC & DI）

控制反转（IoC）与依赖注入（DI）是现代开发中常用的设计模式，特别是在构建可扩展的应用时：

  • IoC：通过控制反转，类之间不直接依赖具体的实例，而是依赖接口或抽象类，由容器进行管理和注入。

  • DI：通过依赖注入，类通过构造函数或 setter 方法获得所需的依赖。

4.ORM（对象关系映射）

使用 ORM 进行数据库操作时，应遵循以下规范：

  • 模型类命名：每个表对应一个模型，类名使用大驼峰命名法。例如，user_profile表的模型类应该命名为 UserProfile。

  • 避免直接使用原生 SQL：尽量通过 ORM 提供的查询构建器或 ActiveRecord 来构建查询，避免 SQL 注入风险。

  • Eager Loading：尽量使用 Eager Loading 来避免 N+1 查询问题。

5. 编码和命名规范

  • 命名规范：
    • 类名使用 PascalCase（大驼峰命名法）。
    • 方法名使用 camelCase（小驼峰命名法）。
    • 常量名使用全大写字母，单词之间用下划线分隔。
    • 变量命名简洁而有意义(通俗易懂）。
  • 代码规范：
    • 每个方法做一件事，避免函数过长。
    • 确保代码注释简洁明了，尤其是复杂的逻辑部分。
    • 使用 PSR-1, PSR-2, PSR-4 等编码规范。

6. 代码自测

  • 在本地环境中充分的自测
  • 尽量覆盖所有测试路径
  • 衡量改动代码的影响范围，不要引发其他bug
  • 考虑是否修复历史数据

其他编程中注意要点：

  • 提交前检查一遍函数的参数是否跟注释一致
  • sql方法中find查询不能直接转toArray()，需要先empty判空
  • 不能用and|or，需要改为&& | ||
  • $laneID = isset($output['laneID']) ? (int)$output['laneID'] : 0;类似这种的要改成$laneID = (int)$output['laneID']
  • empty($laneID) ? 0 : $laneID 要改成 !empty($laneID) ? $laneID : 0
  • 循环时，如果不用键值，则不要在foreach里面写$index => $value，禁止用$i等无意义的变量名
  • 如果只是字符串统一用单引号，如果字符串中引用了变量则统一用双引号，并且需要将变量用{}括起来，主要排查链接参数问题
  • 减少单次变量的使用 \ 禁止硬编码，比如1,0 这种
  • foreach中不要遍历查数据库（先全部查出来再遍历）
